Technical Foundation

Architecture Review

Stack analysis across backend, browser client, native client, and database layers.

B+

Backend: Fastify + WebSocket

Clean separation of concerns. Shared contracts, persistence abstraction, proper auth with scrypt+salt. Needs route modularization and rate limiting.

C+

Browser: Three.js Client

Functional 3D rendering with glTF loading and build tools. Single 1,844-line file with no module structure. Still branded as "ThirdLife."

B-

Native: Godot 4 Client

Full API integration with 3D rendering, build mode, and parcel overlays. 1,297 lines in a single script - needs scene decomposition.

B

Database: PostgreSQL

Auto-creating schema, foreign keys, graceful in-memory fallback. No migrations system, no connection pooling config.

A-

Contracts: Shared Types

Clean TypeScript discriminated unions. Runtime type guards. Good API contract discipline. Some duplication with persistence types.

A-

Real-Time: WebSocket

Solid region-based presence with sequence ordering. Clean peer lifecycle. Needs interest management and reconnection logic.

Recommended Architecture Evolution

Current:   Monolith Fastify + In-Process WebSocket + Optional Postgres
           |-- Single server.ts (1,083 lines)
           |-- Single store.ts (950 lines)
           |-- Single persistence.ts (2,180 lines)

Phase 1:   Modularized Monolith (Weeks 1-4)
           |-- Route plugins (auth, regions, social, economy, admin)
           |-- Service layer (auth.service, region.service, etc.)
           |-- Repository layer (account.repo, region.repo, etc.)
           |-- Auth middleware + request validation

Phase 2:   Scalable Services (Weeks 5-12)
           |-- Redis for sessions + pub/sub
           |-- Region worker processes
           |-- CDN for static assets
           |-- pgBouncer connection pooling

Phase 3:   Production Ready (Weeks 13-20)
           |-- Container orchestration
           |-- CI/CD pipeline
           |-- Monitoring + alerting
           |-- Load testing + benchmarks

What's Done Right

  • TypeScript strict mode throughout
  • Shared contract types between server and client
  • scrypt password hashing with per-account salts
  • Timing-safe password comparison
  • Session expiry with sliding window
  • Build permission system with parcel awareness
  • Dual persistence (memory + Postgres)
  • Asset sync checks between web and native
  • Duplicate function detection in Godot scripts
  • Consistent HTTP status code usage

What Needs Attention

  • All routes in a single 1,083-line file
  • Persistence layer is 2,180 lines in one file
  • No request validation schemas
  • No auth middleware (token checked per-route)
  • Browser client is a single 1,844-line file
  • Godot client is a single 1,297-line script
  • No error logging/monitoring framework
  • No graceful shutdown handling
  • No health check beyond basic GET
  • No environment variable validation

Community Design

MMORPG & Social Features

Deep social systems inspired by 20+ years of MMO design. People don't stay for the game - they stay for the people.

Presence System

  • Online - actively in a region
  • Busy - in build mode or AFK
  • Away - idle 10+ minutes
  • Invisible - online but hidden
  • Custom status messages
  • Friend location sharing

Chat Channels

  • /local - avatars within 20m
  • /region - everyone in region
  • /whisper - private 1:1
  • /group - guild chat
  • /trade - marketplace
  • /help - community support

Trust & Reputation

  • Newcomer - 0-24 hours played
  • Resident - 24h+, verified
  • Trusted - 100h+, good standing
  • Creator - marketplace publisher
  • Mentor - opted-in helper
  • Trust unlocks features progressively

Progression Tracks

  • Builder - Novice to Architect (5 ranks)
  • Social - Newcomer to Community Leader
  • Explorer - Tourist to World Walker
  • Creator - Tinkerer to Legendary
  • Each rank unlocks cosmetics and tools
  • Non-competitive, self-paced

Risk Assessment

Security & Risk Analysis

Critical security issues and mitigation strategies.

Severity Issue Impact Mitigation
CRITICAL No rate limiting on auth Brute force attacks on login Add @fastify/rate-limit, IP-based throttle
CRITICAL CORS allows all origins Cross-site request exploitation Whitelist specific origins per environment
HIGH Token in body, not header Token logging, caching exposure Migrate to Authorization: Bearer header
HIGH No chat message sanitization XSS via chat in browser client HTML-escape all user content before rendering
MEDIUM No CSRF protection Forged requests from malicious sites Add CSRF tokens for state-changing routes
MEDIUM UUID session tokens Slightly predictable format Use crypto.randomBytes(32) for tokens
LOW Admin token in env var Potential leak in logs/config Acceptable for dev, use secrets manager in prod

Already Done Right

  • scrypt password hashing (strong KDF)
  • Per-account random salts
  • Timing-safe password comparison
  • Session TTL with sliding expiry
  • Admin role separation
  • Parcel-based build permissions
  • Object ownership verification

Needs Immediate Attention

  • Rate limiting (all endpoints)
  • CORS origin restriction
  • Input sanitization (chat, names)
  • Token transport (body to header)
  • WebSocket message size limits
  • File upload validation (when added)
  • SQL injection audit (string interpolation)
