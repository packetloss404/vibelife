VibeLife Developer Manual

Complete technical reference for building, extending, and understanding the VibeLife platform — a Minecraft Paper server plugin system with Fastify sidecar and Fabric client mod.

Sidecar Backend

TypeScript Fastify 5, Node.js ESM, 50+ services, 40+ route files, dual-mode persistence (PostgreSQL or in-memory)

Paper Plugin

Java 21 Paper server plugin (Gradle). Bridges Minecraft server events to the sidecar via HTTP/REST. Handles auth, parcels, chat, achievements, economy.

Fabric Mod

Java 21 Fabric client mod (Fabric Loom). Renders UI overlay screens for economy, social, marketplace, achievements, and events.

Communication

REST Plugin → Sidecar via HTTP/REST. Mod → Sidecar via HTTP/REST. Plugin → Mod via plugin messaging channel (vibelife:main).

Authentication & Sessions

MC Auth Flow

Authentication is driven by Minecraft UUIDs. When a player joins the Paper server, the plugin calls the sidecar to create or resume a session.

Player joins MC Server | v Paper Plugin (LoginListener) | +-- POST /api/auth/mc-login -----------> Sidecar | { mcUuid, username } | Lookup or create account | | Create session (8h TTL) |<------ { token, session } -------------+ | +-- Send token to Fabric Mod via | plugin messaging (vibelife:main) | v Fabric Mod receives token | +-- Uses token for all REST calls | GET /api/currency/balance?token=X | GET /api/friends?token=X | etc. v
Note: There is no guest/register/login flow from a sidebar UI. The Minecraft server authenticates the player, and the Paper plugin bridges that identity to the sidecar. The sidecar trusts the plugin's mc-login call because it comes from the server process.

Session Management

TypeScript
// Session TTL: 8 hours
const SESSION_TTL_MS = 1000 * 60 * 60 * 8;

// Session structure
type Session = {
  token: string;          // UUID v4
  accountId: string;      // UUID v4
  avatarId: string;       // UUID v4
  displayName: string;
  mcUuid: string;         // Minecraft UUID
  role: "resident" | "admin";
  expiresAt: number;      // Date.now() + TTL
};

// Stored in global Map
const sessions = new Map<string, Session>();

// Token validation refreshes TTL
export function resolveToken(token: string): Session | null {
  const session = sessions.get(token);
  if (!session) return null;
  if (session.expiresAt < Date.now()) {
    sessions.delete(token);
    return null;
  }
  session.expiresAt = Date.now() + SESSION_TTL_MS; // Refresh
  return session;
}

Token Resolution in Routes

Tokens can be passed in three ways. Routes extract them using a consistent pattern:

TypeScript
// In route handler:
const token = request.body?.token
  ?? request.query?.token
  ?? request.headers.authorization?.replace("Bearer ", "");

const session = resolveToken(token);
if (!session) return reply.status(401).send({ error: "Invalid or expired token" });

Permission System

Build Permission

TypeScript
type BuildPermission = {
  allowed: boolean;
  parcel: Parcel | null;
  reason?: string;
};

export function getBuildPermission(
  regionId: string, accountId: string, x: number, z: number
): BuildPermission {
  const parcel = findParcelAt(regionId, x, z);
  if (!parcel)
    return { allowed: false, parcel: null, reason: "No parcel here" };
  if (parcel.tier === "public")
    return { allowed: true, parcel };
  if (parcel.ownerAccountId === accountId)
    return { allowed: true, parcel };
  if (parcel.collaboratorAccountIds.includes(accountId))
    return { allowed: true, parcel };
  if (!parcel.ownerAccountId)
    return { allowed: false, parcel, reason: "Claim this parcel first" };
  return { allowed: false, parcel, reason: "Owned by another player" };
}

Plugin Messaging Channel

The Paper plugin and Fabric mod communicate through the vibelife:main plugin messaging channel. This is used for:

  • Token handoff: After mc-login, the plugin sends the session token to the Fabric mod so it can make authenticated REST calls.
  • Notifications: The plugin can push real-time notifications (achievement unlocked, currency received, friend request) to the mod for UI display.
  • Sidecar URL: The plugin sends the configured sidecar base URL so the mod knows where to make REST calls.
Java
// PluginMessageHandler.java (Paper side)
public class PluginMessageHandler implements PluginMessageListener {
    public static final String CHANNEL = "vibelife:main";

    public void sendToken(Player player, String token, String sidecarUrl) {
        ByteArrayDataOutput out = ByteStreams.newDataOutput();
        out.writeUTF("TOKEN");
        out.writeUTF(token);
        out.writeUTF(sidecarUrl);
        player.sendPluginMessage(plugin, CHANNEL, out.toByteArray());
    }

    public void sendNotification(Player player, String type, String json) {
        ByteArrayDataOutput out = ByteStreams.newDataOutput();
        out.writeUTF("NOTIFY");
        out.writeUTF(type);
        out.writeUTF(json);
        player.sendPluginMessage(plugin, CHANNEL, out.toByteArray());
    }
}

Switching Modes

TypeScript
export function createPersistence(mode: "memory" | "postgres"): PersistenceLayer {
  if (mode === "postgres") {
    return createPostgresPersistence();  // PostgreSQL (production)
  }
  return createMemoryPersistence();       // In-memory Maps (dev)
}

Adding a New Route

TypeScript
// src/routes/fishing.ts
import { FastifyInstance } from "fastify";
import { resolveToken } from "../world/_shared-state.js";
import { castLine, getFishingSpots } from "../world/store.js";

export function registerFishingRoutes(app: FastifyInstance) {

  // GET /api/fishing/spots?regionId=X
  app.get("/api/fishing/spots", async (request, reply) => {
    const { regionId } = request.query as { regionId: string };
    return getFishingSpots(regionId);
  });

  // POST /api/fishing/cast
  app.post("/api/fishing/cast", async (request, reply) => {
    const { token, x, z } = request.body as { token: string; x: number; z: number };
    const session = resolveToken(token);
    if (!session) return reply.status(401).send({ error: "Unauthorized" });

    const result = castLine(session.token, session.regionId, x, z);
    return result;
  });
}

Architecture Patterns Summary

Sidecar + Plugin Model

The sidecar (Fastify/TS) owns all business logic and state. The Paper plugin bridges MC events to the sidecar via REST. The Fabric mod provides UI overlays.

Barrel Re-exports

store.ts re-exports all services. Routes import from store.js or _shared-state.js only.

REST-Only Communication

No WebSockets. The Paper plugin and Fabric mod both communicate with the sidecar via HTTP/REST. Plugin messaging channel handles token handoff and notifications between plugin and mod.

Dual Persistence

Same interface for memory and PostgreSQL. Switch via env var. Memory mode for dev, PostgreSQL for production.

Permission Layering

Build permissions checked at service level before mutation. Public parcels are always open. The Paper plugin calls the sidecar to check permissions before allowing block placement.

Contract-First Types

All types defined in contracts.ts. This is the single source of truth for the API contract between sidecar, plugin, and mod.
