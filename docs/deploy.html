PacketCraft Deployment Guide

Everything you need to run PacketCraft on your Paper server

📜

Requirements

Before you begin, make sure your server meets these prerequisites.

☕
Java 21+
Required by Paper 1.21.x. Use Eclipse Temurin or GraalVM.
📄
Paper 1.21.1+
PacketCraft targets the Paper API. Spigot and Bukkit are not supported.
🌐
Node.js 20+
Powers the Fastify sidecar service. Install via nvm or your package manager.
🗃
PostgreSQL
Recommended for production. An in-memory store is available for testing only.
🚀

Quick Start

Follow these four steps to go from zero to a running PacketCraft server.

1

Set up PostgreSQL

Create a dedicated database and user for PacketCraft. Adjust the password for production environments.

sudo -u postgres psql -c "CREATE USER packetcraft WITH PASSWORD 'packetcraft';" sudo -u postgres psql -c "CREATE DATABASE packetcraft OWNER packetcraft;"
2

Start the Sidecar

The Fastify sidecar provides the REST API that the plugin communicates with. Point it at your database.

cd /opt/packetcraft DATABASE_URL=postgres://packetcraft:packetcraft@127.0.0.1:5432/packetcraft npm run dev
Warning: After starting, check http://localhost:3000/api/health — it must show "persistence":"postgres". If it shows "memory", the database connection failed silently and you are running on the in-memory store. Your data will be lost on restart.
3

Build the Plugin

Compile the Paper plugin using Gradle. The shadow JAR bundles all dependencies.

cd paper-plugin ./gradlew shadowJar

The output JAR will be at paper-plugin/build/libs/packetcraft-paper-*.jar.

4

Install & Start Paper

Copy the plugin JAR into your Paper server's plugins directory and launch the server.

cp paper-plugin/build/libs/packetcraft-paper-*.jar paper-server/plugins/ cd paper-server && java -Xms1G -Xmx2G -jar paper.jar --nogui

On first boot, PacketCraft will generate its default configuration at plugins/PacketCraft/config.yml.

⚙

Configuration

PacketCraft is configured through two places: the plugin config file and sidecar environment variables.

Plugin Config — plugins/PacketCraft/config.yml

SettingDefaultDescription
sidecar.url http://localhost:3000 Base URL of the Fastify sidecar API. Change this if the sidecar runs on a different host or port.
sidecar.api-key changeme Shared secret between the plugin and sidecar. You must change this in production.
sidecar.timeout-ms 5000 HTTP request timeout in milliseconds. Increase if your sidecar is on a remote host.
channel packetcraft:main Plugin messaging channel used to communicate with the Fabric client mod.
regions — Maps Minecraft world names to PacketCraft region IDs. Example: world: region-overworld.

Sidecar Environment Variables

VariableDefaultDescription
PORT 3000 The port the Fastify sidecar listens on.
DATABASE_URL — PostgreSQL connection string. If unset, falls back to in-memory storage.
CORS_ORIGINS * Comma-separated list of allowed CORS origins. Restrict this in production.
ADMIN_BOOTSTRAP_TOKEN — If set, allows bootstrapping the first admin account via the API.

Communication Flow

When a player joins the server, the following happens:

  • The Paper plugin detects the join event and calls the sidecar at /api/auth/mc-login
  • The sidecar authenticates the player and returns a session token
  • The plugin sends the token to the player's Fabric mod via the packetcraft:main plugin messaging channel
  • The Fabric mod can now make authenticated requests directly to the sidecar for UI data
🗃

Database

The sidecar manages all database operations. There are no manual migrations to run.

Auto-Created Tables

On first startup with a valid DATABASE_URL, the sidecar automatically creates all required tables. No migration commands are needed — just point it at an empty database and go.

Backups

Use standard PostgreSQL tools to back up your data:

pg_dump -U packetcraft -h 127.0.0.1 packetcraft > packetcraft-backup-$(date +%Y%m%d).sql

Schedule this with cron for regular automated backups.

Silent Fallback Warning: If the DATABASE_URL is missing or the connection fails, the sidecar silently falls back to an in-memory store. Everything will appear to work, but all data is lost on restart. Always verify persistence via the /api/health endpoint.
🗺

Parcel Cache

The plugin maintains a local cache of parcel data to minimize API calls and keep protection checks fast.

How It Works

  • On startup, the plugin performs an initial sync 2 seconds after enable to let the sidecar finish booting
  • After that, parcel data is re-synced from the sidecar every 5 minutes
  • Protection checks (block place/break, interact) are resolved against the local cache, so they are instant
  • When a player claims or modifies a parcel, the cache is updated immediately in addition to the API call
💰

Vault Economy Integration

PacketCraft registers as a Vault economy provider, making it compatible with any plugin that uses the Vault API.

Details

  • Soft dependency: Vault is not required. If Vault is not installed, the economy still works through PacketCraft commands
  • Currency: Vibes (integer-based, no decimals)
  • Standard Vault API: Other plugins can call getBalance(), depositPlayer(), withdrawPlayer(), etc.
  • Balances are stored in the sidecar database, not in Vault's config
🔐

Permissions

packetcraft.admin

Grants the ability to bypass parcel protection. Players with this permission can build and break blocks in any parcel, regardless of ownership.

Default: op — only server operators have this permission by default. Assign it via your permissions plugin (e.g., LuckPerms) as needed.

🎨

Optional: Fabric Mod

The Fabric client mod gives players a rich in-game UI. It is optional — the server works fine without it.

Build & Distribute

cd fabric-mod ./gradlew build

Distribute the resulting JAR to your players. They place it in their .minecraft/mods/ folder alongside Fabric Loader.

Connection

The mod connects to localhost:3000 by default (the sidecar). On a real server, the player's mod receives its auth token from the plugin via plugin messaging — no manual configuration is needed on the client side.

Monitoring & Health

Keep an eye on your PacketCraft deployment with these built-in tools.

Health Endpoint

The sidecar exposes GET /api/health which returns the current status, including:

  • status — should be "ok"
  • persistence — should be "postgres" in production
  • uptime — seconds since sidecar start
curl -s http://localhost:3000/api/health | jq .

Plugin Logs

All plugin log messages are prefixed with [PacketCraft]. Look for these in your Paper server console or logs/latest.log.

  • Startup: Confirms sidecar connection, parcel sync count, Vault hook
  • Warnings: Sidecar unreachable, API errors, fallback behavior
  • Errors: Authentication failures, database issues relayed from the sidecar
✅

Production Checklist

Before going live, make sure you have addressed every item on this list.

Pre-Launch Checklist

  • Change sidecar.api-key from the default changeme to a strong, random secret
  • Set a strong PostgreSQL password (not packetcraft)
  • Verify /api/health shows "persistence":"postgres"
  • Configure CORS_ORIGINS to only allow your domain(s)
  • Set up automated database backups with pg_dump and cron
  • Monitor server logs for [PacketCraft] warnings and errors
  • Configure regions in config.yml to map your world names to region IDs
  • Test Vault integration if you use other economy plugins
  • Run the sidecar behind a process manager (systemd, pm2) for auto-restart
